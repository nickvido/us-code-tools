# GitHub-safe section anchors and separated subsection paragraphs

## Summary
Update the markdown renderer so chapter/embed output no longer emits visible `{#...}` anchor syntax in section headings, and so structured subsection headings render as separate markdown paragraphs instead of running together in one block. This preserves navigability in embedded title/chapter markdown while matching GitHub-flavored Markdown rendering behavior.

## Context
- `src/transforms/markdown.ts` currently appends `{#anchor}` to embedded section headings when `renderSectionBody()` receives an `anchor` option.
- GitHub-flavored Markdown renders `{#section-8}` literally, so embedded section headings show implementation text to readers.
- `renderSubsectionHeading()` currently returns a single bold inline line, and `renderContentNodes()` only inserts a blank line when the previous line is not labeled; consecutive subsection headings therefore collapse into one paragraph in rendered GitHub output.
- The repository already has markdown renderer unit tests in `tests/unit/transforms/markdown.test.ts` and snapshot coverage in `tests/unit/transforms/__snapshots__/markdown.test.ts.snap`.
- This issue is limited to markdown rendering behavior; no parser, schema, CLI, or output path changes are required.

## Acceptance Criteria

### Functional
#### 1. Embedded section anchor rendering
- [ ] `renderSectionHeading()` or equivalent embedded-section heading logic must not emit the literal substring ` {#` anywhere in generated markdown output. <!-- Touches: src/transforms/markdown.ts, tests/unit/transforms/markdown.test.ts -->
- [ ] When `renderChapterMarkdown()` or `renderUncategorizedMarkdown()` renders embedded sections, each section that previously received `embeddedSectionAnchor(section.sectionNumber)` must still emit an addressable anchor immediately before its heading, using HTML anchor markup of the form `<a id="section-<normalized-id>"></a>` on its own line. <!-- Touches: src/transforms/markdown.ts, tests/unit/transforms/markdown.test.ts -->
- [ ] Standalone section markdown generated by `renderSectionMarkdown()` must continue to render the top-level heading as `# § {sectionNumber}. {heading}` without adding any HTML anchor line. <!-- Touches: src/transforms/markdown.ts, tests/unit/transforms/markdown.test.ts, tests/unit/transforms/__snapshots__/markdown.test.ts.snap -->

#### 2. Structured subsection paragraph separation
- [ ] When `renderContentNodes()` renders consecutive `subsection` nodes with `structuredSubsectionHeadings: true`, the markdown output must contain a blank line before each subsection heading after any preceding body text or preceding subsection block, so GitHub renders each subsection as its own paragraph. <!-- Touches: src/transforms/markdown.ts, tests/unit/transforms/markdown.test.ts -->
- [ ] A rendered structured subsection heading with body text must keep the existing bold label/heading format (`**(a) Heading** body text` or `**(a)** body text` when no heading exists) while being separated by blank lines from surrounding content. <!-- Touches: src/transforms/markdown.ts, tests/unit/transforms/markdown.test.ts, tests/unit/transforms/__snapshots__/markdown.test.ts.snap -->
- [ ] Nested child lines under a subsection (paragraphs, subparagraphs, clauses, items, and continuation text) must retain their current source order and indentation after subsection paragraph separation is introduced. <!-- Touches: src/transforms/markdown.ts, tests/unit/transforms/markdown.test.ts -->

#### 3. Regression coverage
- [ ] Unit or snapshot tests must cover chapter/embed rendering for a section with an embedded anchor and assert that the output contains `<a id="section-..."></a>`-style markup and does not contain `{#section-...}`. <!-- Touches: tests/unit/transforms/markdown.test.ts -->
- [ ] Unit or snapshot tests must cover at least two sibling subsection nodes and assert that the rendered markdown contains `\n\n**(a)` and `\n\n**(b)` boundaries (or equivalent paragraph breaks) rather than a single continuous paragraph. <!-- Touches: tests/unit/transforms/markdown.test.ts -->
- [ ] Existing renderer tests for standalone section output, deep hierarchy ordering, and cross-title links must continue to pass without expectation changes unrelated to the new anchor/paragraph formatting. <!-- Touches: tests/unit/transforms/markdown.test.ts, tests/unit/transforms/__snapshots__/markdown.test.ts.snap -->

### Non-Functional
- [ ] Performance: Rendering the existing markdown test fixtures must remain deterministic and complete within the current unit test runtime budget; no new network or filesystem I/O may be introduced into renderer functions. <!-- Touches: src/transforms/markdown.ts -->
- [ ] Compatibility: Output must be valid GitHub-flavored Markdown/HTML mix accepted by repository markdown viewers, using only inline-safe HTML anchor tags and no custom markdown extensions. <!-- Touches: src/transforms/markdown.ts -->

## Out of Scope
- Changing the USLM parser or `embeddedSectionAnchor()` normalization rules.
- Reformatting paragraph, subparagraph, clause, or item nodes that are not structured subsection headings.
- Updating historical generated markdown artifacts outside the normal renderer/test workflow.
- Adding browser-side behavior, JavaScript, or CSS for anchors.

## Dependencies
- Existing markdown renderer in `src/transforms/markdown.ts`.
- Existing unit test harness in `tests/unit/transforms/markdown.test.ts`.
- Existing normalization helper `embeddedSectionAnchor()` from `src/domain/normalize.ts`.

## Acceptance Tests (human-readable)
1. Run the markdown renderer unit tests.
2. Render embedded/chapter markdown for a fixture or test section with `sectionNumber: '8'`.
3. Verify the embedded output contains a standalone line `<a id="section-8"></a>` immediately before `## § 8. ...`.
4. Verify the embedded output does not contain `{#section-8}` anywhere.
5. Render a section body with chapeau text followed by sibling subsection nodes `(a)` and `(b)` under `structuredSubsectionHeadings: true`.
6. Verify the output contains a blank line before `**(a)** ...` and another blank line before `**(b)** ...`.
7. Verify nested child lines under subsection `(a)` still appear after `(a)` in the same relative order and with the same indentation as before.
8. Verify standalone `renderSectionMarkdown()` output still starts directly with `# § ...` and has no prepended HTML anchor line.

## Edge Case Catalog
- Empty anchor input: if no embedded `anchor` option is supplied, heading rendering must not add an HTML anchor placeholder or blank extra line.
- Malformed subsection data: subsections with label only, heading only, text only, or empty child arrays must still render a single subsection block without duplicate blank lines.
- Consecutive subsection nodes after unlabeled chapeau text: each subsection must begin a new paragraph.
- Consecutive subsection nodes with nested continuation text after children: continuation lines must remain attached to the correct subsection content order.
- Mixed node sequences: paragraph → subsection → paragraph → subsection must not collapse any subsection heading into an adjacent paragraph.
- Delimiter/encoding cases: subsection text containing quotes, em dashes, Unicode symbols, emoji, or RTL text must not break the blank-line separation logic.
- Duplicate/normalized section numbers: anchor rendering must continue using the existing normalized ID derived from `embeddedSectionAnchor()` for values such as `36B`, `2/3`, or zero-padded numbers.
- Degraded behavior: because rendering is pure in-process code, there is no network/database/cache fallback path; failures should remain ordinary test/runtime exceptions rather than silent output degradation.

## Verification Strategy
- **Pure core:** `renderSectionHeading()`, `renderContentNodes()`, `renderStructuredLine()`, and `renderSubsectionHeading()` should remain pure string/array transformation functions.
- **Properties:** (1) Embedded output never contains literal `{#...}` fragments. (2) Standalone section output never gains an HTML anchor line. (3) Every structured subsection block after the first content block is preceded by exactly one blank separator in compacted output. (4) Child node ordering and indentation remain stable.
- **Purity boundary:** I/O is limited to test fixture reads in the test suite; markdown renderer functions themselves perform no filesystem, network, or process I/O.
